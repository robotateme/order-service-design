# ADR-001: Использование Outbox Pattern для публикации событий заказа

## Статус

Accepted

## Контекст

Сервис оформления заказа сохраняет заказ в своей базе данных и должен публиковать интеграционные события во внешний брокер сообщений. Простая схема "сначала commit в БД, потом publish в брокер" создает окно отказа:

- заказ уже сохранен, но событие может не быть отправлено;
- при повторной попытке возможны дубли;
- при нескольких publisher-процессах нужна координация конкурентной обработки.

Для сервиса важна надежная доставка событий о создании и изменении заказа во внешние системы.

## Решение

Использовать `Outbox Pattern`:

- доменные события создаются внутри агрегата;
- application layer преобразует их в интеграционные события;
- бизнес-данные и запись в `outbox_events` сохраняются в одной транзакции;
- отдельный `OutboxPublisher` читает неопубликованные записи и отправляет их в брокер;
- публикация выполняется с retry-механизмом и техническими полями для блокировки и диагностики.

## Последствия

Плюсы:

- атомарность записи бизнес-данных и события;
- устойчивость к сбоям брокера;
- управляемый retry-flow;
- более предсказуемая интеграция с внешними сервисами.

Минусы:

- дополнительная таблица и фоновый процесс;
- необходимость очистки старых записей outbox;
- возможна повторная доставка, поэтому consumer должен быть идемпотентным.

## Альтернативы

1. Публикация напрямую из application service после commit.
Недостаток: событие теряется при падении между commit и publish.

2. Two-phase commit / distributed transaction.
Недостаток: высокая сложность и плохая практическая применимость для большинства брокеров.

3. CDC из таблиц бизнес-данных.
Недостаток: слабее контролируется бизнес-смысл событий и сложнее управлять контрактами сообщений.
