# Проектирование сервиса оформления заказа

Этот репозиторий содержит набор UML-диаграмм для сервиса оформления заказа в интернет-магазине. Проект описывает архитектуру приложения, структуру данных, основной сценарий создания заказа и интеграционный паттерн `domain events + outbox`.

## Что описывает модель

Сервис поддерживает:

- создание заказа через HTTP-контроллер;
- проверку товаров через внешний `ProductService`;
- сохранение заказа и его позиций;
- выбор типа доставки: самовывоз или доставка по адресу;
- выбор типа оплаты: карта или кредит.
- генерацию доменных событий внутри агрегата заказа;
- маппинг доменных событий в интеграционные;
- надежную публикацию событий через outbox-таблицу с retry и блокировкой.

## Архитектурная идея

Модель построена вокруг слоистой структуры:

- `Interface` принимает входящий запрос;
- `Application` обрабатывает команду создания заказа;
- `Domain` содержит сущности заказа, доставки, оплаты и доменные события;
- `Infrastructure` отвечает за сохранение данных, outbox, брокер-клиент и публикацию событий.

Основной поток выглядит так:

1. Клиент отправляет запрос на создание заказа.
2. Контроллер формирует `CreateOrderCommand`.
3. Обработчик валидирует товары, доставку и оплату.
4. Агрегат `Order` создает `OrderCreatedDomainEvent`.
5. Обработчик извлекает доменные события и преобразует их в интеграционные.
6. Заказ, позиции, доставка, платеж и outbox-событие сохраняются в одной транзакции.
7. Клиент получает успешный ответ.
8. Фоновый publisher блокирует пачку событий из outbox и публикует их во внешний брокер.
9. При ошибке публикации увеличивается `retry_count`, сохраняется ошибка и назначается повторная попытка.

## Модель данных

В БД предусмотрены основные сущности:

- `orders` — заказ;
- `order_items` — позиции заказа;
- `deliveries` — общая запись о доставке;
- `delivery_pickup` — данные самовывоза;
- `delivery_address` — адресная доставка;
- `payments` — общая запись об оплате;
- `payment_credit` — данные кредитной оплаты.
- `outbox_events` — очередь интеграционных событий для гарантированной публикации.

Такой подход позволяет хранить общие атрибуты в базовых таблицах, выносить специализированные поля в отдельные таблицы по типу доставки или оплаты и отдельно фиксировать события, которые должны быть опубликованы во внешние системы. `outbox_events` моделируется как общая таблица сервиса и связывается с бизнес-сущностями через `aggregate_type` и `aggregate_id`, а не через жесткий FK только на `orders`.

В `outbox_events` дополнительно предусмотрены технические поля:

- `message_id` — уникальный идентификатор сообщения для идемпотентности;
- `status` — состояние записи (`pending`, `processing`, `published`, `failed`);
- `retry_count`, `next_retry_at`, `error_message` — поддержка повторных попыток;
- `locked_by`, `locked_at` — координация нескольких publisher-процессов;
- `headers_json` — метаданные интеграционного сообщения.

## Domain Events и Outbox

В модели используется стандартный подход для надежной интеграции:

- агрегат `Order` создает доменное событие `OrderCreatedDomainEvent`;
- обработчик команды сохраняет бизнес-данные и запись в `outbox_events` атомарно;
- `IntegrationEventMapper` преобразует внутреннее событие домена во внешний контракт;
- отдельный `OutboxPublisher` выбирает и блокирует неопубликованные события;
- перед публикацией запись переводится в `processing` и фиксирует `locked_by`/`locked_at`;
- после успешной отправки в брокер запись помечается как опубликованная и освобождает блокировку;
- при ошибке запись не теряется, освобождает блокировку и переходит в повторную обработку.

Это нужно, чтобы не терять события в ситуации, когда бизнес-транзакция уже закоммичена, а отправка в брокер еще не произошла или завершилась ошибкой. Отдельный `message_id` и явный retry-flow также уменьшают риск дублей и делают поведение системы предсказуемым при сбоях.

## Consumer и Inbox

Для end-to-end надежности добавлен сценарий обработки на стороне consumer:

- consumer читает событие из брокера;
- `messageId` сначала фиксируется в inbox-хранилище;
- если сообщение уже обрабатывалось, оно пропускается как дубликат;
- если сообщение новое, выполняется бизнес-обработка и запись помечается как обработанная.

Это закрывает вторую половину задачи идемпотентности: outbox снижает риск потери события у producer, inbox снижает риск повторной обработки у consumer.

## Жизненный цикл заказа

Отдельная state-диаграмма показывает ключевые статусы заказа:

- `Draft` — заказ еще формируется;
- `PendingPayment` — заказ создан и ожидает оплаты;
- `Paid` — оплата подтверждена;
- `ReadyForPickup` / `Shipped` — заказ передан в сценарий выдачи;
- `Completed` — заказ успешно получен;
- `Cancelled` — заказ отменен на допустимом этапе.

## Компоненты системы

Компонентная диаграмма показывает основные runtime-части решения:

- `HTTP API` принимает внешние запросы;
- `Command Handlers` координируют use case;
- `Domain Model` содержит бизнес-правила;
- `Repositories` работают с `PostgreSQL`;
- `Outbox Publisher` публикует интеграционные события в брокер;
- `Product Service` используется для валидации товаров.

## Развертывание

Диаграмма развертывания показывает базовую инфраструктурную схему:

- клиент обращается к `Order Service API`;
- API и `Outbox Publisher Worker` развернуты отдельно;
- бизнес-данные и outbox хранятся в `PostgreSQL`;
- события публикуются во внешний `Message Broker`;
- валидация товаров идет через отдельный `Product Service`.

## Как открыть диаграммы

Диаграммы написаны в формате PlantUML. Их можно открыть:

- в IDE с плагином PlantUML;
- через онлайн-рендерер PlantUML;
- локально через CLI.

Пример локального рендера:

```bash
plantuml architecture.puml component.puml consumer-flow.puml database.puml deployment.puml order-state.puml sequence.puml
```

После этого рядом с `.puml` файлами будут созданы изображения диаграмм.
