# Версионирование интеграционных событий

## Цель

Документ описывает, как менять контракты интеграционных событий без неконтролируемых поломок consumer-ов.

## Базовые правила

- каждое событие имеет `eventType` и `schemaVersion`;
- несовместимые изменения требуют новой версии схемы;
- совместимые изменения допускаются только как back-compatible расширения;
- consumer должен быть готов игнорировать неизвестные поля.

## Что считается совместимым изменением

- добавление нового необязательного поля;
- добавление новых header-ов;
- расширение набора допустимых значений, если consumer к этому готов.

## Что считается несовместимым изменением

- удаление существующего поля;
- переименование поля;
- изменение типа поля;
- изменение семантики поля без смены версии.

## Рекомендуемый подход

1. Сохранять `eventType` стабильным для одной логической сущности события.
2. Передавать версию в `headers.schemaVersion` или отдельном поле.
3. При несовместимом изменении публиковать новую версию контракта параллельно со старой на переходный период.
4. Удалять старую версию только после подтвержденной миграции consumer-ов.

## Пример

- `OrderCreatedIntegrationEvent` + `schemaVersion = 1.0`
- `OrderCreatedIntegrationEvent` + `schemaVersion = 1.1` для совместимого расширения
- `OrderCreatedV2IntegrationEvent` или новая schema version для несовместимого контракта
